# SC0001 Manifest Audit — BL-CAM-0

## Purpose

Before adding structured `camera`, `lighting`, and `motion` fields to the beat and shot layers (BL-CAM-1), this audit fixes three decisions that the v2 plan review surfaced:

1. **Scope**: how many clips and beats does SC0001 actually have, and which beats already carry `required_element_ids` (from A7.4e0)?
2. **Truncation policy**: the upstream packer in [scripts/agents/omni_clip_planner.py](../../scripts/agents/omni_clip_planner.py) truncates beat `content` to 120 characters when writing `prompt_action` to manifests. B-EL5c only fixed the downstream prompt YAML. What is the long-term position?
3. **Merge logic**: when the packer merges multiple beats into one shot, which beat's `camera`/`lighting`/`motion` survives?

This document records the answers. No code or schema is modified.

## 2. `prompt_action` truncation policy decision

### Current state

The packer at [scripts/agents/omni_clip_planner.py:163-164](../../scripts/agents/omni_clip_planner.py#L163-L164) truncates beat `content` to 117 characters and appends `"..."` when writing `prompt_action`:

```python
content = beat.get("content", "")
if len(content) > 120:
    content = content[:117] + "..."
```

**9 of the 16 manifest shots across SC0001 have truncated `prompt_action`**: every shot whose source beat content exceeds 120 chars (most of them). B-EL5c fixed `CLIP_SC0001_01` prompt YAML by manually authoring complete prose; the manifest still carries the truncation.

### Options considered

- **A. Keep truncation in manifest, complete prose in adapter**. Adapter reads `scene_beat_plan.source_beats[].content` for the full text via `source_scene_beat_plan_ref` + `source_beat_ids[0]`. Manifest `prompt_action` stays as a readable summary.
- **B. Remove truncation everywhere**. Raise the 120-char limit (or remove it), regenerate all 8 manifests, then adapter consumes full text directly.
- **C. Add `prompt_action_full` field**. Optional shot-level field for the complete text; truncated `prompt_action` stays for backward compatibility.

### Decision: Option A

**Rationale**:
1. **No backward break**. Manifest readers (operators reading YAML, prompt review tools, the critic) keep getting a glanceable summary.
2. **No 8-manifest regeneration just for ellipses**. BL-CAM-1b will regenerate manifests anyway to add `camera`/`lighting`/`motion`, but that regeneration changes new fields, not `prompt_action` content.
3. **Adapter already has the right footing**. `KlingOmniAdapter.generate_from_clip_manifest()` already receives `source_scene_beat_plan_ref` in the manifest. Adding a `_resolve_full_action(beat_id)` helper is local and well-scoped.
4. **Beat plan is canonical**. The beat plan is the single source of truth for narrative content. Letting the adapter reach back to it (rather than duplicating the full text in manifests) reduces drift risk.

**Operational consequence**: KO-CAM-1 must include the beat-plan re-lookup. The trigger heuristic is `prompt_action.endswith("...")` — when true, fetch the beat from `source_scene_beat_plan_ref` and use its full `content`.

**Tests required in KO-CAM-1** (already enumerated in plan):
- `test_adapter_reads_beat_full_text_when_action_truncated`
- A regression that adapter still works when `prompt_action` is non-truncated (CLIP_05 shots provide this case).

## 3. Merge logic decision for `camera` / `lighting` / `motion`

The packer in [scripts/agents/omni_clip_planner.py:175-263](../../scripts/agents/omni_clip_planner.py#L175-L263) has three merge paths: backward (short_insert into preceding regular beat), forward (orphan inserts into next regular beat), and trailing (trailing inserts into last shot). The A7.4e0 `_union_elements` helper at lines 175-179 unions `required_element_ids` across all merged beats.

For the new cinematography fields, union is wrong. A short insert that uses a different camera framing should not silently override the anchor shot's setup.

### Decision

| Field | Merge rule |
|---|---|
| `camera` | **Anchor beat only**. In backward merge: the host (non-insert) beat's `camera`. In forward merge: the next non-insert beat's `camera` (insert beats are absorbed forward into the host). In trailing merge: the existing shot's `camera` is preserved; trailing inserts contribute none. |
| `lighting` | **Anchor beat only**. Same anchor-beat policy as `camera`. |
| `motion.subject_intensity` | **Max** across all merged beats. Conservative: a short high-intensity insert (e.g., a wrist pulse-check) raises the merged shot's signaled motion intensity to ensure Kling renders both anchor and insert action correctly. |
| `motion.camera_intensity` | **Max** across all merged beats. Same logic. |
| `motion.blocking_notes` | **Anchor beat only**. Insert beats' blocking notes are discarded to keep the merged shot direction coherent and deterministic. |

### Why anchor-only for camera/lighting and max for motion

- **Camera**: shot framing/angle/movement is the cinematographer's overall setup. A 2-second wrist insert merging into a 5-second cabinet inventory shot should not reframe the 7-second result; it stays in the cabinet shot's framing.
- **Lighting**: same reasoning — lighting setup is global to the shot.
- **Motion intensity**: this is what tells Kling how aggressively to animate. A high-intensity insert means the shot *will* contain high-intensity motion at some point, so the signal must propagate. Taking max is the conservative choice.
- **Blocking notes**: human-authored prose. Concatenating insert and host notes produces awkward, redundant prompts. Anchor-only preserves authorial intent.

### Helper design for BL-CAM-1

Mirror [scripts/agents/omni_clip_planner.py:175-179](../../scripts/agents/omni_clip_planner.py#L175-L179) `_union_elements`:

```python
def _max_motion(shots: list[_Shot]) -> dict:
    """Across merged shots, take max for intensity fields, anchor for notes."""
    if not shots:
        return {}
    out = {}
    sub = max((s.motion.get("subject_intensity") or 0.0) for s in shots)
    cam = max((s.motion.get("camera_intensity") or 0.0) for s in shots)
    if sub > 0.0:
        out["subject_intensity"] = sub
    if cam > 0.0:
        out["camera_intensity"] = cam
    notes = shots[0].motion.get("blocking_notes")
    if notes:
        out["blocking_notes"] = notes
    return out
```

For `camera` and `lighting`, no helper is needed — the anchor's dict is reused verbatim. The packer just doesn't change those fields when absorbing insert beats.

## Side-effect to flag in BL-CAM-1b

When BL-CAM-1b regenerates all 8 SC0001 manifests via the packer:

1. **`required_element_ids` will be added** to manifests 02–08 at both clip-level (union of shots) and shot-level (per beat). This is desired — it completes the A7.4e0 work that previously only touched CLIP_01.
2. **`prompt_action` will remain unchanged** under truncation policy A — the packer's 120-char truncation is preserved.
3. **`camera`, `lighting`, `motion`** blocks will appear on shots that have beat-level metadata in the updated beat plan.

The PR description for BL-CAM-1b will explicitly call out these three side-effects so review can verify nothing unintended changed.
